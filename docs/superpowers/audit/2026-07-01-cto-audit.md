# CTO Audit Report: Playmorrow

**Date:** 2026-07-01
**Auditor:** CTO (automated deep audit)
**Version:** 0.1.0 (pre-Beta)
**Repository:** `playmorrow` — Monorepo (pnpm workspaces + Turborepo)

---

## Executive Summary

Playmorrow is a well-architected monorepo with a clear separation of concerns: a NestJS REST API (`apps/api`), a Next.js 15 App Router frontend (`apps/web`), and shared packages (`packages/database`, `packages/types`, `packages/config`). The schema is mature with 25+ models, and the codebase shows deliberate engineering decisions (session-based auth, XP/gamification system, notification engine, OAuth flows).

**However, several critical issues must be resolved before Beta launch:** hardcoded/fake data leaks in multiple production pages, mock client routing that could serve fake data in production, missing indexes on high-query tables, no CI/CD pipeline, no backup strategy, and several security concerns around token exposure in URLs and missing rate limiting on mutation endpoints.

**Overall Score: 62/100** — Functionally impressive but not yet production-safe.

## 5. Security

### 5.1 Session Management
- Session cookie: `httpOnly: true`, `secure: true` in production, `sameSite: 'none'` in production
- ⚠️ `sameSite: 'none'` requires Secure context, but cookie is set correctly — acceptable for cross-origin API
- Sessions stored as SHA-256 hash, 7-day expiry
- Auth versioning: `authVersion` incremented on password reset, sessions checked against it
- Account lockout: 5 failed attempts → 15-minute lock

### 5.2 OAuth Flow
- Google + GitHub OAuth implemented via Passport.js
- ⚠️ No CSRF state parameter in OAuth flow (`apps/api/src/auth/oauth/oauth.controller.ts`)
- ⚠️ OAuth callback sets session cookie then redirects with tokens in URL (`?email=...&displayName=...`) — these could leak via Referer header
- Callback catches errors broadly and redirects to login with generic message

### 5.3 CSRF Protection
- **NONE.** No CSRF tokens anywhere. Session-based auth relies on `SameSite` cookie attribute.
- `SameSite: 'none'` in production + `credentials: 'include'` on fetch = vulnerable if any subdomain is compromised
- Mitigation: Use `SameSite: 'lax'` as default, limit `sameSite: 'none'` to auth-specific endpoints

### 5.4 XSS Prevention
- Helmet.js used with CSP directives
- CSP allows `'unsafe-inline'` for scripts and styles — necessary for Next.js but reduces protection
- CSP `connectSrc: ["'self'", "*"]` — overly permissive, allows connections to any origin
- No server-side output sanitization for user-generated content (devlog body, comments)

### 5.5 SQL Injection
- Prisma parameterized queries prevent SQL injection — ✅

### 5.6 File Upload Security
- MIME type validation | ✅
- Magic byte validation | ✅
- 10MB file size limit | ✅
- Images only (JPEG, PNG, GIF, WebP, SVG) | ✅
- ⚠️ SVG upload allowed — SVG can contain JavaScript if served with `image/svg+xml` content type
- No file type whitelist by extension (using regex on filename)
- No virus/malware scanning
- Files stored on local disk (not S3/CDN) — not production-scalable

### 5.7 Permission Leaks
- `GET /studios/:slug/members` returns all members including user IDs — acceptable for public profiles
- `GET /me/feed` — properly guarded with SessionAuthGuard
- `GET /me/notifications/stream` — authentication via query parameter `?token=...` — **token exposed in URLs/logs**

### 5.8 Additional Concerns
- ⚠️ `notifications.controller.ts:98` — SSE stream accepts JWT as query parameter `?token=...` — this leaks the JWT into server logs, browser history, and referrer headers
- ⚠️ No HTTPS enforcement in code (relies on platform/reverse proxy)
- No API key validation for service-to-service calls
- Refresh token rotation implemented but old token is revoked, not the entire family
